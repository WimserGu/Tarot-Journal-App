# Tarot Journal App 产品简报

> 文档状态：初版产品基础
> 当前阶段：需求验证，暂不进入功能扩张
> 核心假设：当塔罗记录按“长期议题 → 固定问题 → 多次记录”组织时，用户会比使用聊天记录、备忘录或纸张更容易持续记录并发现变化。

## 1. 一句话产品定位

Tarot Journal App 是一款面向持续进行塔罗自我观察的用户、按长期议题和固定问题组织记录，并用历史对比与基础统计帮助用户识别变化和重复模式的私人日记工具。

它首先是记录与复盘工具，而不是单次解牌工具，也不提供确定性的未来预测。

## 2. 核心目标用户

### 主要目标用户

经常使用实体塔罗牌，并会围绕关系、学业、事业或个人成长等长期议题重复提问的个人用户。

这类用户通常具备以下特征：

- 每周至少记录一次塔罗结果，或希望建立稳定的记录习惯。
- 目前使用聊天记录、备忘录、表格或纸质笔记，资料较分散。
- 在意同一问题随时间发生的变化，而不只关心一次抽牌的解释。
- 愿意自己记录牌面和感受，希望工具帮助整理，而不是替自己下结论。

### 暂非核心用户

- 只想快速获得一次 AI 解牌、不打算保存历史的人。
- 主要寻找社交、占卜师服务或公开分享社区的人。
- 需要复杂专业牌阵管理、客户管理或商业接单功能的职业占卜师。

## 3. 用户最重要的三个痛点

1. **记录分散，无法形成连续上下文。** 同一长期议题的提问、牌面、解读和后续发展散落在多个地方，用户很难回看完整过程。
2. **重复记录成本高，数据也不一致。** 用户每次都要重新输入问题和牌阵信息，容易遗漏日期、正逆位、位置或后续反馈，之后无法可靠比较。
3. **主观印象难以验证。** 用户可能感觉某张牌“经常出现”，却无法快速确认实际频率，也很难看出花色、正逆位和个人反馈随时间的变化。

## 4. 产品的三个核心价值

1. **让记录围绕长期议题持续积累。** 以“议题 → 固定问题 → 多次记录”为主结构，把零散牌阵变成可追踪的个人档案。
2. **让记录和复盘足够轻松。** 复用固定问题和牌阵位置，快速录入实体牌，并允许稍后补充个人解读与后续反馈。
3. **让模式建立在真实历史数据上。** 提供时间线、重复牌、花色分布和正逆位比例；所有总结注明数据范围，不把统计关联表述成因果或预言。

## 5. 一个典型用户的一天

林是一名正在写论文的研究生。她已经在 App 中创建了“论文进展”议题，并保存了固定问题：“今天最值得优先处理的阻碍是什么？”

早上，她用实体牌抽取一张牌，打开该议题并选择固定问题。日期自动填入，她选择牌名和正逆位，写下一句个人解读，约一分钟完成记录。

下午工作结束后，她回到当天记录，补充实际发生的事情和自己的反馈。她不需要重新输入牌面，也不会丢失早上的思考。

周末，她打开该问题的历史页面，按日期查看过去几周的记录。系统显示重复出现的牌、花色分布和正逆位比例，并明确提示当前样本量。她结合自己的解读和后续反馈，判断哪些主题确实反复出现，而不是让 App 替她预测论文结果。

## 6. MVP 必须包含的功能

### A. 私人账户与数据隔离

- 简单注册、登录和退出。
- 每位用户只能访问自己的议题、问题和记录。
- 日记数据默认不公开。

### B. 长期议题管理

- 创建、编辑、归档议题。
- 议题至少包含名称和可选说明。
- 首页能看到进行中的议题，并快速进入最近使用的议题。

### C. 固定问题管理

- 在议题下创建、编辑、停用固定问题。
- 可记录计划频率，例如每日、每周或按需；MVP 只保存频率，不负责推送提醒。
- 问题可复用于新的记录，避免重复输入。

### D. 实体牌记录

- 从固定的标准 78 张塔罗牌列表中选择牌名。
- 每张牌记录正位或逆位，并可填写牌阵位置。
- 每条记录包含问题、日期、一个或多个牌位、个人解读和可稍后补充的后续反馈。
- 保存前校验必填内容，避免产生无法比较的残缺记录。

### E. 历史浏览与同题比较

- 按议题和固定问题查看历史记录。
- 同一问题的记录按时间排列，能清楚看到日期、牌面、正逆位、个人解读和后续反馈。
- 支持打开和编辑单条历史记录。

### F. 基础统计

- 显示记录数量和统计所覆盖的日期范围。
- 显示重复牌及出现次数。
- 显示大阿尔卡那与四种花色的分布。
- 显示正位与逆位比例。
- 统计可按单个固定问题查看；不在 MVP 中生成 AI 结论。

### G. 基础可靠性

- 数据保存成功或失败时给出明确反馈。
- 关键删除操作需要二次确认。
- 对空数据和样本量过少提供清楚的状态说明。

## 7. MVP 暂时不做的功能

- AI 解牌、AI 聊天或自动趋势叙述。
- App 内随机抽牌、洗牌动画或沉浸式牌桌。
- 推送通知、连续打卡奖励和游戏化机制。
- 图片识牌、相机扫描或手写识别。
- 社区、公开分享、关注、评论或占卜师市场。
- 自定义牌组、不同体系的牌义库和牌面图片上传。
- 复杂图表、跨议题相关性分析、预测或因果判断。
- 付费订阅、多语言、桌面端和网页端。
- 多人协作、客户管理或职业占卜师工作台。
- 导入聊天记录、备忘录或历史纸质记录的自动迁移工具。

这些功能只有在核心记录与复盘行为被验证后，才重新评估优先级。

## 8. 衡量 MVP 是否成功的指标

建议先邀请 10–20 名符合目标特征的测试用户，进行 4 周封闭测试。重点衡量是否形成核心行为，而不是下载量。

| 指标           | 定义                                                                   | 初步成功标准 |
| -------------- | ---------------------------------------------------------------------- | ------------ |
| 激活率         | 注册后 24 小时内完成“创建议题 + 创建固定问题 + 保存首条记录”的用户比例 | ≥ 70%        |
| 核心行为达成率 | 测试期内，针对同一固定问题在至少 3 个不同日期完成记录的用户比例        | ≥ 50%        |
| 复盘使用率     | 已拥有至少 3 条同题记录的用户中，打开过该问题历史或统计页面的比例      | ≥ 60%        |
| 记录完成率     | 开始创建后成功保存的记录占比                                           | ≥ 85%        |
| 记录效率       | 一张牌的常规记录从开始到保存所需时间中位数                             | ≤ 2 分钟     |
| 第 4 周留存    | 第 1 周完成首条记录、且第 4 周仍至少新增或复盘一次记录的用户比例       | ≥ 35%        |
| 主观价值       | 测试结束后认同“比原来的记录方式更容易比较长期变化”的用户比例           | ≥ 70%        |

最关键的判断标准是“核心行为达成率 + 复盘使用率”。如果用户愿意记录，却不查看历史和统计，说明产品可能只是另一个数字笔记本，核心差异尚未成立。

## 9. 可能存在的产品风险

| 风险                       | 可能影响                           | MVP 应对方式                                                             |
| -------------------------- | ---------------------------------- | ------------------------------------------------------------------------ |
| 手动录牌步骤过多           | 用户尝试几次后放弃记录             | 复用固定问题、自动填充日期、提供牌名搜索，并允许稍后补充长文本           |
| 历史数据过少               | 统计结果没有价值，或被用户过度解读 | 始终显示样本量和日期范围；样本不足时只展示事实，不生成“趋势”结论         |
| 高频抽牌加重焦虑或重复确认 | 产品可能强化不健康的使用习惯       | 不使用催促式打卡，不鼓励短时间对同一问题反复抽牌；文案强调反思与记录边界 |
| 私密内容泄露               | 用户失去信任，造成严重隐私风险     | 默认私密、按 `user_id` 隔离数据，并在数据库层启用访问控制策略            |
| 卡牌与牌组差异             | 不同牌组命名或体系导致录入困难     | MVP 先支持标准 78 张牌及通用名称，不承诺覆盖所有塔罗体系                 |
| 统计被理解为预测或因果     | 偏离产品定位并误导用户             | 使用“出现次数”“分布”等事实性语言，避免“将会”“意味着结果”等确定性表达     |
| 功能范围持续扩大           | 延迟发布，难以验证真正的核心价值   | 每个新增需求都先判断是否直接帮助“持续记录”或“同题复盘”                   |

## 10. 建议的最小可验证版本

### 要验证的唯一核心假设

用户是否会因为能够按长期议题和固定问题积累记录，并查看同题历史与简单统计，而持续使用该产品替代原有的零散记录方式？

### 最小功能边界

最小可验证版本只包含以下闭环：

1. 用户登录。
2. 创建一个长期议题。
3. 在议题下创建一个固定问题。
4. 录入实体牌结果，包括日期、牌名、正逆位、可选牌阵位置和个人解读。
5. 稍后为记录补充后续反馈。
6. 查看该问题按时间排列的历史记录。
7. 在至少 3 条记录后，查看重复牌、花色分布和正逆位比例。

建议首轮进一步限制为：只支持标准 78 张塔罗牌、每条记录 1–3 张牌、一个固定问题对应多次记录。暂不加入 AI、App 内抽牌、提醒和复杂图表。

### 测试方式

- 招募 10–20 名目前已经在重复记录塔罗结果的用户。
- 连续测试 4 周，要求用户使用自己的实体牌和真实长期议题。
- 通过匿名事件统计观察创建、保存、复盘行为，并在第 2 周和第 4 周进行简短访谈。
- 重点询问：录入是否比原方式省力、同题历史是否帮助复盘、基础统计是否可信且易懂。

### 继续开发的判断

如果核心行为达成率和复盘使用率达到第 8 节的标准，则进入完整 MVP，并优先改善记录效率和历史比较体验。如果用户不愿持续记录，应先解决录入负担或议题结构问题，而不是增加 AI 或抽牌功能。

## 当前产品边界

第一版的最短价值路径是：

**创建议题 → 复用固定问题 → 记录实体牌 → 补充反馈 → 比较历史 → 查看事实统计**

任何不能直接缩短这条路径、提高记录完整性或增强同题复盘价值的功能，都不应进入首版。
